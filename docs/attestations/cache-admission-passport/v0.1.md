# SemWitness Cache Admission Passport predicate v0.1

Status: SemWitness alpha interoperability contract

TypeURI:

```text
https://github.com/aantenore/semwitness/blob/main/docs/attestations/cache-admission-passport/v0.1.md
```

This document defines the `v0.1` compatibility family of the SemWitness Cache
Admission Passport predicate. It uses an
[in-toto Statement v1](https://github.com/in-toto/attestation/blob/main/spec/v1/statement.md)
to carry content-free lineage for one SemWitness intent-cache shadow
qualification. It does not authorize cache admission, lookup, storage, or
serving.

The TypeURI is controlled by the SemWitness repository. Compatible additions
remain in this `v0.1` family and follow in-toto's monotonic extension rules. An
incompatible predicate requires a new TypeURI and an explicit migration.

## Required Statement profile

The Statement MUST have exactly one supported subject and this predicate shape:

```json
{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [
    {
      "name": "semwitness-intent-cache-shadow-qualification",
      "digest": {
        "sha256": "<64 lowercase hexadecimal characters>"
      }
    }
  ],
  "predicateType": "https://github.com/aantenore/semwitness/blob/main/docs/attestations/cache-admission-passport/v0.1.md",
  "predicate": {
    "artifact": {
      "id": "semwitness-cache-admission-passport",
      "version": "0.1.0"
    },
    "profile": "intent-plan-read",
    "authentication": "none",
    "decision": "shadow-qualified",
    "activationCeiling": "shadow-only",
    "basis": {
      "schema": "semwitness.dev/intent-cache-shadow-qualification/v1alpha1",
      "artifact": {
        "id": "semwitness-intent-cache-shadow-qualifier",
        "version": "1"
      },
      "provenance": "host-attested-unsigned",
      "evidenceAuthentication": "none",
      "producerIdentity": null
    },
    "validity": {
      "notBefore": "1970-01-01T00:00:00.000Z",
      "notAfter": "1970-01-01T00:00:00.001Z",
      "revocationId": "hmac-sha256:revocation:<64 lowercase hexadecimal characters>"
    },
    "scope": {
      "deploymentScopeDigest": "sha256:<64 lowercase hexadecimal characters>",
      "cacheNamespace": "hmac-sha256:cache-namespace:<64 lowercase hexadecimal characters>",
      "tenant": "hmac-sha256:tenant:<64 lowercase hexadecimal characters>",
      "domain": "hmac-sha256:intent-domain:<64 lowercase hexadecimal characters>",
      "operation": "hmac-sha256:operation:<64 lowercase hexadecimal characters>"
    },
    "contracts": {
      "cacheAdmissionPolicyDigest": "sha256:<64 lowercase hexadecimal characters>",
      "normalizationPolicyDigest": "sha256:<64 lowercase hexadecimal characters>",
      "dependenciesDigest": "sha256:<64 lowercase hexadecimal characters>"
    },
    "evidence": {
      "reportDigest": "sha256:<64 lowercase hexadecimal characters>",
      "evaluatorDigest": "sha256:<64 lowercase hexadecimal characters>"
    }
  }
}
```

The literals `authentication: "none"`, `decision: "shadow-qualified"`, and
`activationCeiling: "shadow-only"` are protocol invariants. A consumer MUST NOT
interpret `bound: true`, the Statement, or a signature over the Statement as an
active admission decision.

## Derivation

The producer MUST strictly parse one
`semwitness.dev/intent-cache-shadow-qualification/v1alpha1` manifest before
derivation. It MUST NOT accept caller-supplied issuer, approval, authorization,
decision, timestamp, or activation-ceiling overrides.

The qualification artifact is the exact UTF-8 canonical serialization emitted
by SemWitness, with no trailing line feed. The subject `sha256` value is the
SHA-256 digest of those exact bytes, encoded as lowercase hexadecimal without a
`sha256:` prefix.

Every predicate field is derived from the same parsed qualification:

| Predicate field                        | Qualification source or derivation                                                                |
| -------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `basis`                                | Qualification schema, artifact, provenance, evidence authentication, and producer identity        |
| `validity.notBefore` / `notAfter`      | Qualification epoch values converted to canonical RFC 3339 UTC strings with millisecond precision |
| `validity.revocationId`                | Qualification revocation HMAC                                                                     |
| `scope`                                | Deployment digest and cache namespace, tenant, domain, and operation HMACs                        |
| `contracts.cacheAdmissionPolicyDigest` | Qualified cache-admission policy digest                                                           |
| `contracts.normalizationPolicyDigest`  | Qualified normalization-policy digest                                                             |
| `contracts.dependenciesDigest`         | SHA-256 digest of the complete canonical dependency inventory                                     |
| `evidence`                             | Qualified report and evaluator digests                                                            |

`notBefore` and `notAfter` MUST be canonical UTC RFC 3339 strings in the form
`YYYY-MM-DDTHH:mm:ss.sssZ`, and `notBefore` MUST precede `notAfter`. They and
`revocationId` are copied, digest-bound claims. This profile does not enforce a
clock, revocation, expiry, or replay state.

## Serialization and digest semantics

The supported Statement profile MUST be serialized as deterministic canonical
JSON encoded as UTF-8, with no byte-order mark and no trailing line feed. A
Statement file emitted by SemWitness consists of exactly those bytes.

Two digests have intentionally different meanings:

- `canonicalProfileDigest` is the SHA-256 digest of the normalized, supported
  predicate profile after bounded unknown extensions have been removed. It is
  useful for supported-profile comparison only.
- `payloadDigest` is the SHA-256 digest of the exact supplied Statement bytes.
  It is available only when a verifier receives a string or byte payload. This
  is the appropriate digest for an exact-payload receipt or commitment.

A consumer MUST NOT use `canonicalProfileDigest` as a commitment to a signed or
received payload because different extended payloads can normalize to the same
supported profile.

## Parsing, extensions, and binding

A conforming parser MUST enforce bounded strict JSON, reject ambiguity and
unsafe object forms, require exactly one supported subject, and validate all
supported fields. Following in-toto's
[Statement parsing rules](https://github.com/in-toto/attestation/blob/main/spec/v1/statement.md),
bounded data-only unknown fields MAY be ignored when producing the normalized
supported profile.

SemWitness binding is deliberately stricter than parsing. If any unknown field
is present at a supported Statement or predicate level, the inspector MUST set
`extensionsPresent: true` and `bound: false`, even when the normalized profile
matches a freshly derived Statement. This preserves the content-free policy:
standard in-toto fields such as subject `content`, or predicate extensions that
could carry raw prompts, cannot pass binding unnoticed.

For a Statement without extensions, `bound` is true only if its complete
supported canonical profile equals a freshly derived Statement for the
separately supplied qualification. When the verifier receives a string or byte
payload, those exact bytes MUST also equal the canonical supported
serialization: alternative whitespace, key order, or escapes produce
`canonicalPayload: false` and `bound: false`. Object-only verification has no
byte identity, reports `canonicalPayload: null`, and compares the supported
profile only. Binding proves deterministic equality, not producer identity,
evidence honesty, current validity, or authorization.

## DSSE use

SemWitness v0.1 does not create or trust signatures or envelopes. If another
system wraps the exact Statement payload in
[DSSE](https://github.com/secure-systems-lab/dsse/blob/master/protocol.md), it
SHOULD use `application/vnd.in-toto+json` as `payloadType` as described by the
[in-toto envelope specification](https://github.com/in-toto/attestation/blob/main/spec/v1/envelope.md).

DSSE signs the Pre-Authentication Encoding `PAE(payloadType, payload)`, thereby
authenticating both the payload type and the exact payload bytes. A verifier
MUST verify that PAE and signature before passing the same decoded payload bytes
to the Passport parser. It MUST NOT parse and reserialize before signature
verification, and MUST NOT substitute `canonicalProfileDigest` for the exact
payload.

Even a valid signature authenticates only the signed Statement under the
verifier's external trust policy. It does not repair
`evidenceAuthentication: "none"`, prove that evidence was held out, enforce
time or revocation, or raise `activationCeiling: "shadow-only"`.

## Confidentiality and operations

The Passport contains no prompt, response, Intent IR, candidate, plan, value,
credential, provider payload, URL, path, or raw error. It is nevertheless
sensitive metadata. Stable HMACs and digests reveal equality and may disclose
tenant relationships or workload shape to anyone who can compare artifacts.

Implementations MUST protect emitted files with owner-only permissions such as
`0600`, refuse unsafe overwrite and symlink behavior, and apply deployment ACLs
and retention controls. Operators SHOULD NOT publish Passport Statements or
qualification manifests as ordinary build artifacts or logs.

## Conformance requirements

An implementation of this profile MUST test:

- deterministic, exact qualification and Statement bytes without a trailing
  line feed;
- subject binding and mutation of every derived field;
- canonical RFC 3339 validity conversion and rejection;
- strict JSON and resource limits;
- bounded monotonic parsing plus strict content-free extension rejection during
  binding;
- non-canonical string/byte payload rejection during binding;
- the distinction between exact `payloadDigest` and extension-eliding
  `canonicalProfileDigest`;
- data-only isolation, privacy snapshots, private no-clobber output, package
  export, and installed CLI/plugin execution.
