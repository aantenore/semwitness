# SemWitness Cache Admission Decision predicate v0.1

Status: SemWitness alpha interoperability contract

TypeURI:

```text
https://github.com/aantenore/semwitness/blob/v0.5.0-alpha.5/docs/attestations/cache-admission-decision/v0.1.md
```

The release tag in this URI is the immutable protocol identity. It MUST never
be moved or reused; semantic changes require a new artifact version and
TypeURI. Publishing any package or plugin that emits this profile requires the
exact `v0.5.0-alpha.5` tag to exist first and the URI above to resolve to this
document.

This document defines the `v0.1` compatibility family for a deterministic
[in-toto Statement v1](https://github.com/in-toto/attestation/blob/main/spec/v1/statement.md)
about one exact SemWitness Cache Admission Passport payload and one exact,
eligible `CacheHitWitness` payload. The Statement preserves per-decision
lineage without carrying a raw prompt, Intent IR, plan, cache value, credential,
or raw identifier.

The Statement is unsigned evidence. It does not authorize a lookup, cache
write, value delivery, canary, or active runtime action.

## Required Statement profile

A supported Statement MUST have exactly the following two named subjects. The
pair has joint semantics; neither subject independently grants any permission.

## Input and cross-link requirements

The producer MUST derive the Statement from separate evidence. It MUST NOT
accept caller-supplied decision fields, subjects, issuer, signer, approval,
timestamp, nonce, authorization result, or serving authority.

Creation MUST fail closed unless all of the following hold:

1. The Passport is an exact canonical UTF-8 payload without a BOM or trailing
   line feed, contains no extensions, and binds to the supplied canonical
   shadow qualification.
2. The `CacheHitWitness` is an exact canonical UTF-8 payload, its internal
   digest verifies against the separate `NormalizationWitness`, and its only
   decision is `eligible`, `applied: false`, `CACHE_HIT_ELIGIBLE`.
3. Entry and lookup bindings are both `plan` plus `read` and match on the
   admission, freshness, scope, policy, and normalization fields enforced by
   the SemWitness witness verifier.
4. The operation binding is internally valid and matches the qualified domain,
   operation, intent digest, effect, operation registry, and ontology.
5. The normalization witness's normalizer and ontology match the qualified
   artifacts. The plan's operation-registry, planner, and tool-registry digests
   match the qualification dependency inventory.
6. The entry-source binding is internally valid and matches the admitted entry
   and value digests. Its source HMAC intentionally describes the source that
   created the stored entry; it need not equal the current paraphrase's source
   HMAC.
7. Namespace, tenant, admission policy, and normalization policy match the
   Passport and qualification. The operation-registry, planner, and
   tool-registry dependencies are cross-linked to the hit. The full dependency
   inventory and deployment-scope digest are qualification declarations copied
   from the exact Passport; v0.1 does not claim that the hit witness binds
   every qualified dependency or the deployment-scope digest.
8. The candidate value's exact bytes hash to the witness value digest. Those
   bytes never enter the Statement.
9. The cache key and keyed commitments are recomputed with a deployment-owned
   secret containing at least 256 bits generated by a CSPRNG. A human
   passphrase or predictable 32-byte pattern is non-conformant. Commitments are
   never accepted from the caller.

The profile records historical shadow evidence only. These checks do not prove
that an upstream producer was honest, that the hit was produced under every
dependency or deployment attribute declared by the qualification, or that the
evidence reflects the current authoritative runtime state.

## Exact subject and commitment derivation

The Passport subject is SHA-256 over the exact supplied canonical Passport
bytes. It MUST NOT use the Passport's extension-eliding
`canonicalProfileDigest`. The witness subject is SHA-256 over the exact supplied
canonical `CacheHitWitness` bytes. In-toto subject digest values omit the
`sha256:` prefix.

`cacheKeyDigest` is the existing SemWitness deployment-keyed HMAC over the
complete canonical cache binding. Entry and value commitments use the same
validated deployment secret with a separate protocol prefix and domain:

```text
HMAC-SHA256(
  secret,
  "semwitness.dev/intent-cache-artifact-commitment/v1\0" ||
  ("entry" | "value") || "\0" ||
  canonicalCacheBinding || "\0" ||
  (entryDigest | valueDigest)
)
```

The output is encoded as `hmac-sha256:cache-entry:<hex>` or
`hmac-sha256:cache-value:<hex>`. Domain separation makes entry and value
commitments non-interchangeable. Keying prevents a public dictionary oracle for
low-entropy values, but commitments remain linkable wherever the same secret,
binding, and digest are reused.

The deployment secret MUST be generated by a cryptographically secure random
generator, stored in a secret manager, scoped to the deployment, and rotated
under an explicit external key epoch. v0.1 does not serialize that key epoch;
operators must retain the epoch-to-secret association for the desired evidence
retention window and must not silently verify old Statements with a new key.

## Serialization, parsing, and binding

The supported profile is canonical JSON encoded as UTF-8, without a BOM or
trailing line feed. Strict parsing rejects duplicate keys, malformed UTF-8,
unsafe numbers, resource-limit violations, proxies, accessors, symbols, hidden
properties, custom prototypes, sparse arrays, and unsupported subject shapes.

Bounded, data-only unknown fields MAY be ignored when parsing, following
in-toto's monotonic extension rule. SemWitness binding is stricter:

- `profileBound` is true only when the extension-free supported profile equals
  a freshly derived Statement for all supplied evidence;
- `canonicalPayload` is true only when the exact supplied Statement bytes equal
  the canonical supported serialization;
- `bound` is true only when `profileBound` and `canonicalPayload` are both true
  and no extension is present.

Object-only verification has no byte identity. It can report
`profileBound: true`, but MUST report `canonicalPayload: null`,
`payloadDigest: null`, and `bound: false`.

`canonicalProfileDigest` identifies the normalized supported profile.
`payloadDigest` identifies exact supplied Statement bytes. A signature,
transparency record, or external commitment MUST use the original payload bytes
or `payloadDigest`, never the extension-eliding profile digest.

## Deliberate v0.1 limits

This unsigned profile does not enforce a clock, Passport expiry, revocation,
authorization freshness, key epoch, nonce, sequence, idempotency ledger, or
replay store. It deliberately contains no timestamp or caller-provided issuer.
Recreating or replaying the same Statement cannot cause serving because the
embedded ceiling remains `shadow-only`, `applied` remains false, and
`servingAuthority` remains `none`.

A future active profile requires a separate authenticated approval envelope,
current authorizer, trusted clock and revocation verifier, key-epoch contract,
and atomic replay store. Those properties require a new protocol family; they
MUST NOT be inferred from or added in place to v0.1.

## External envelopes and transparency

SemWitness v0.1 does not create or trust signatures. A future external
[DSSE](https://github.com/secure-systems-lab/dsse/blob/master/protocol.md) or
[Sigstore Bundle](https://docs.sigstore.dev/about/bundle/) adapter can
authenticate the exact payload using `application/vnd.in-toto+json`. It MUST
verify `PAE(payloadType, payload)` before parsing or reserializing the payload.
A valid signature authenticates bytes under an external trust policy; it cannot
raise the embedded shadow-only ceiling or repair unauthenticated source
evidence.

This artifact is intentionally named a Decision Statement, not a SCITT Receipt.
[SCITT RFC 9943](https://www.rfc-editor.org/rfc/rfc9943.html) and
[COSE Receipts RFC 9942](https://www.rfc-editor.org/rfc/rfc9942.html) define
transparency-service registration and inclusion evidence. Such a receipt can
prove that bytes were registered; it does not prove this cache decision was
correct or authorize value delivery.

## Confidentiality and operations

The Statement excludes raw content and identifiers but is still sensitive
metadata. Stable HMACs, digests, subject identity, and timing of file creation
can expose equality and workload shape. Implementations MUST keep Statement and
private evidence files behind deployment ACLs, avoid stdout/log echoes of
secrets or values, and apply retention controls. The bundled writer uses
owner-only mode `0600` plus no-clobber semantics on POSIX. On Windows, where
POSIX mode bits do not define the DACL, operators MUST provision and verify an
owner-restricted parent directory ACL; no-clobber and symlink refusal still
apply.

## Conformance requirements

An implementation MUST test deterministic golden bytes, exact subject digests,
all cross-links above, HMAC domain separation, canonical versus non-canonical
payloads, extension handling, invariant escalation, value substitution,
privacy snapshots, strict bounded parsing, private file safety, package exports,
and installed CLI/plugin execution on supported platforms.
