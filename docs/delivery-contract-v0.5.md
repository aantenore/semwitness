# Delivery contract: verified request preparation

- Date: 2026-07-15
- Target release: `v0.5.0-alpha.1`
- Maturity: experimental AI-enabled product
- Delivery mode: pull request, then prerelease only after required CI is green

## Outcome

SemWitness will expose an opt-in host boundary that can place a mechanically
verified candidate into the request an application actually sends to its model.
The first adapter targets AI SDK `LanguageModelV4Middleware` and only considers
explicitly selected text tool results. It does not create a gateway, own a
provider call, or turn IntentWitness into an active semantic cache.

## Scope

### Must

- Keep shadow/identity behavior as the default.
- Keep the provider-neutral request preparer free of network clients,
  credentials, retries, routing, model selection, and provider SDK types.
- Require an `apply-verified` codec policy, exact tokenizer evidence, and a
  strict promotion manifest before returning transformed content.
- Bind promotion to the exact policy, tokenizer fingerprint, codec versions,
  held-out corpus, evaluation report, adapter artifact, provider/model,
  reviewed prompt/tool contracts, and normalized selector/trust map.
- Require zero unsafe accepts, zero measured task-quality regressions, and at
  least 10% median net input-token savings in the bound evaluation.
- Fall back to the original byte-identical text on codec, verification,
  promotion, storage, retrieval, decoding, or adapter failure.
- Reject JavaScript strings that cannot round-trip losslessly through UTF-8.
- Limit active delivery in this alpha to `tool`-role, typed JSON transformed by
  `json-jcs@1`; decoder-dependent codecs remain shadow-only.
- Keep evidence content-free: identifiers, digests, counters, decisions, and
  bounded reason codes only.
- Bound messages, tool parts, selected candidates, and total preparation time;
  a timeout or overflow must roll back the entire transformation to the exact
  original call options without emitting partial decisions.
- Snapshot proof and candidate evidence privately across asynchronous verifier
  calls; verifier mutation must not alter an admitted or returned proof.
- Prove through a wrapped mock model that the actual downstream call receives a
  shorter canonical JSON tool result while every protected or unselected part
  remains unchanged.

### Should

- Make AI SDK selection declarative and strict rather than auto-detecting
  arbitrary prose.
- Preserve immutable input objects and provider options.
- Keep package dependency direction `ai-sdk adapter -> host -> existing core`.

### Out of scope

- Lossy prose or instruction compression.
- Rewriting system, developer, user, assistant reasoning, tool schemas, tool
  calls, error results, files, or multimodal content.
- Serving cached plans, observations, or responses from IntentWitness.
- A transparent OpenAI-compatible proxy, a new cache, or a vector database.
- Provider calls, authentication, rate limiting, multi-tenancy, streaming
  recovery, or response-body transformation.
- Claiming savings for output tokens already generated by a model.
- Transparent Codex interception. A Codex plugin or hook may run an explicit
  workflow, but SDK/App Server ownership is required to replace `turn/start`
  input before the model request.

## Requirements and acceptance

| ID  | Requirement             | Acceptance evidence                                                                                                                                                               |
| --- | ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | Default-safe activation | Missing promotion, shadow policy, heuristic tokenizer, or unpromoted codec returns the exact original and a bounded bypass reason.                                                |
| R2  | Promotion provenance    | Strict schema rejects extra fields and binds policy, tokenizer, codec, corpus/report evidence, deployment-scope digest, held-out status, safety counts, and median net savings.   |
| R3  | Host isolation          | The host module imports no AI SDK or provider package and performs no network I/O.                                                                                                |
| R4  | Narrow AI SDK mapping   | Only configured AI SDK v4 `tool` message `tool-result` text parts are candidates; provider/model and normalized selector/trust bindings must match the promoted deployment scope. |
| R5  | Fail-open availability  | Exceptions, timeout, work-budget overflow, and malformed/non-JSON inputs preserve the exact original call options and never prevent the provider call.                            |
| R6  | Downstream proof        | A mock wrapped model captures the transformed request; canonical JSON is shorter and parses to the same typed value.                                                              |
| R7  | Regression safety       | Existing test suite, format, lint, typecheck, build, package dry-run, dependency audit, and cross-platform CI pass.                                                               |
| R8  | Atomic evidence         | Host and adapter use private proof/content snapshots; mutations during async verification or multi-candidate preparation cannot yield partial application or inconsistent proof.  |

## Architecture

```text
AI SDK call options
  -> explicit tool-result selector
  -> provider/model + prompt/tool + selector/trust scope digest
  -> host-owned Segment metadata
  -> SemWitness simulate + mechanical verification
  -> promotion-manifest gate
  -> retrieve verified candidate or original
  -> wrapped model call
```

- `semwitness/host` owns promotion validation and one-segment request
  preparation. It depends only on existing SemWitness contracts.
- `semwitness/ai-sdk` maps AI SDK v4 prompt parts to the host boundary. It does
  not choose providers or make calls.
- Existing codec, proof, policy, CAS, and tokenizer ports remain authoritative.
- IntentWitness remains an independent shadow-only bounded context.
- StageFabric remains independent. A future composition root may consume both
  packages without coupling either domain core to the other.

## Threat model delta

- A forged or stale promotion manifest could enable an unevaluated transform.
  The alpha treats the manifest as a trusted host-owned deployment artifact and
  binds all relevant digests. The live provider/model identity is checked by the
  adapter; deriving honest prompt/tool contract digests remains a host duty.
  Signatures remain a future provenance boundary.
- A host could misclassify prose as JSON data. The first adapter exposes only a
  declarative tool-result selector, validates JSON through the existing codec,
  and still requires evaluation evidence for active use.
- Candidate or CAS failure must not break inference. Runtime failures are
  normalized to content-free bypasses and the original is retained in memory.
- A slow or never-settling preparer must not hold the provider call forever.
  Strict scan/candidate ceilings and one total deadline make the whole mapping
  atomic; underlying non-cooperative work may finish later but cannot alter the
  already forwarded original request.
- A verifier may retain and mutate its arguments after resolving. The host keeps
  private proof/candidate snapshots, gives the verifier separate copies, and
  rechecks private evidence after the asynchronous boundary.
- Provider behavior can differ even for typed-equivalent JSON. Mechanical
  equivalence is necessary but not sufficient; held-out task evaluation is a
  mandatory promotion input and remains scoped to its model/prompt/tool
  contract.

## Verification plan

Critical:

- Unit tests for strict promotion parsing, digest binding, exact-tokenizer gate,
  codec allowlist, tampering, threshold failure, and byte-identical fallback.
- Adapter tests for selected/unselected parts, non-JSON input, provider options,
  input immutability, scan/candidate overflow, never-settling preparation,
  atomic timeout rollback, buffered content-free decisions,
  non-authoritative observer failures, and thrown host errors.
- Host tests where a delegated verifier mutates both proof and candidate after a
  valid verification; the admitted proof remains private, coherent, and frozen.
- Wrapped-model integration test for both `doGenerate` and `doStream` parameter
  transformation through the middleware contract where supported.
- Sentinel scan proving reports and errors contain no source or transformed
  content.

Release evidence:

- `pnpm check`
- `pnpm build`
- `pnpm pack --dry-run`
- production dependency audit
- Linux, macOS, and Windows CI from a clean checkout

## Rollout and rollback

The feature is additive and lives behind separate package exports. Existing
callers receive no behavior change. Rollback removes the middleware from the
host composition root; no stored data or schema migration is required. The
original request remains available for every call and is always the fallback.
Late non-cooperative preparer work has no authority over the provider request or
decision stream.
